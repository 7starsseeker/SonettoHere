# 记忆系统

## 两层记忆架构

SonettoHere 的记忆系统分为两层：

```
┌─────────────────────────────────────────────┐
│  短期记忆 (ShortTermMemory)                   │
│  - 当前会话的消息历史                          │
│  - 基于 tiktoken 的自动裁剪                    │
│  - 阈值：28,000 tokens                        │
├─────────────────────────────────────────────┤
│  记忆叙事 (Memory Narrative)                   │
│  - 跨会话的自然语言记忆                         │
│  - 每轮对话后 LLM 增量更新                      │
│  - 存储介质：config/personas/MEMORY.md         │
└─────────────────────────────────────────────┘
```

---

## 第一层：短期记忆（ShortTermMemory）

在 [memory/short_term.py](../memory/short_term.py) 中。

### 核心数据结构

```python
class ShortTermMemory:
    def __init__(self, max_tokens: int = 28000):
        self._store = InMemoryChatMessageHistory()   # LangChain 内置
        self._max_tokens = max_tokens
        self._encoding = tiktoken.encoding_for_model("gpt-4")
```

`InMemoryChatMessageHistory` 是 LangChain 提供的内存级消息历史存储。它本质上是一个 `list[BaseMessage]`，提供 `add_message`、`add_messages`、`clear` 方法。

### Token 自动裁剪机制

```python
def add_message(self, message: BaseMessage) -> None:
    self._store.add_message(message)
    self._trim()          # ← 每次添加消息后自动检查

def _trim(self) -> None:
    msgs = self._store.messages
    while self._count_tokens(msgs) > self._max_tokens and len(msgs) > 2:
        msgs.pop(0)       # ← 从头部删除最旧的消息

def _count_tokens(self, messages: list[BaseMessage]) -> int:
    total = 0
    for m in messages:
        content = m.content if isinstance(m.content, str) else str(m.content)
        total += len(self._encoding.encode(content))
        total += 4         # ← 每条消息的元数据开销估算
    return total
```

### 裁剪策略的要点

1. **FIFO 淘汰**：从消息列表头部（最旧的消息）开始删除，保留最近的对话上下文
2. **保留最少 2 条**：即使超限也保留至少 2 条消息，避免上下文完全丢失
3. **token 估算**：使用 `tiktoken` 的 `gpt-4` 编码器（与 DeepSeek 的 tokenizer 高度相似），每条消息额外加 4 个 token 估算元数据开销
4. **阈值选择**：28,000 tokens 是一个权衡值——DeepSeek Chat 的上下文窗口为 128K，28K 的阈值在保留足够上下文的同时避免了消息列表无限膨胀导致的延迟和成本问题

## 第二层：记忆叙事（Memory Narrative）

在 [memory/narrative.py](../memory/narrative.py) 中。

### 工作原理

每轮对话结束后，将本轮**裸对话消息**（不做任何预处理）连同旧的 `MEMORY.md` 一起送给 LLM，让 LLM 在现有记忆基础上融合新信息，写回 `config/personas/MEMORY.md`。

```
回合结束:
  turn_messages (裸消息)
      │
      └── update_narrative(llm, turn_messages)
            │
            ├── 冷启动 (MEMORY.md 不存在):
            │     turn_messages → LLM → 生成 MEMORY.md
            │
            └── 常态 (MEMORY.md 已存在):
                  turn_messages + 旧 MEMORY.md → LLM → 更新 MEMORY.md
```

### 为什么用裸消息而不是提取的摘要？

旧系统先用 Extractor 把对话压缩成 summary，再拿 summary 去检索或生成。这导致信息损失叠加，且两个模块耦合。叙事模块直接读裸消息，让 LLM 看到原始对话的全部细节。

### 叙事 LLM Prompt

**冷启动**（MEMORY.md 不存在时）：

```
你是一个记忆叙事师。根据以下用户与AI助手的对话，
撰写一份关于这个用户的自然语言记忆。

要求：
1. 用第三人称描述用户
2. 涵盖：用户身份、偏好习惯、重要事件、注意事项
3. 简洁连贯，200-400字
4. 保留关键细节（地址、时间、项目名等）
5. 不确定的信息不要编造
```

**常态更新**（MEMORY.md 已存在时）：

```
你是一个记忆叙事师。以下是当前你对用户的记忆，以及新一轮的对话。
请在现有记忆的基础上，融入新信息，写一份更新后的记忆。

要求：
1. 用第三人称描述用户
2. 涵盖：用户身份、偏好习惯、重要事件、注意事项
3. 简洁连贯，200-400字
4. 保留已有正确信息，新信息优先于旧信息
5. 如果新信息与旧记忆矛盾，以新信息为准
6. 不确定的信息不要编造
```

### 叙事注入系统提示词

在 [agent/prompts.py](../agent/prompts.py) 中，`build_system_prompt()` 将 `MEMORY.md` 的内容注入到 `"## 我对用户的记忆"` 段落。`build_enhanced_prompt()` 每轮重新调用 `build_system_prompt()`，确保叙事始终是最新的。

### 叙事内容示例

```markdown
Miso 是北京邮电大学沙河校区的学生，住在北京市昌平区。
他正在开发 SonettoHere 项目，使用 LangGraph + DeepSeek。
他偏好简洁的回答，不喜欢冗长的解释。
他的空闲时间是周一/三/六/日上午十点、周三/日下午三点等。
他养了一只猫叫年糕。
近期他在处理 QQ Bot 集成的 botpy SDK 日志问题。
他曾提醒：天气查询应直接用天气工具，不要走模糊地址搜索。
```

### 文件存储

| 文件 | 说明 |
|------|------|
| `config/personas/MEMORY.md` | AI 自动生成的叙事，每轮更新，gitignore |
| `config/personas/USER.md` | 用户手动填写的自述，gitignore |
| `config/personas/USER.example.md` | 用户自述模板，git 跟踪 |
| `memory/user_init.py` | 首次运行从 example 复制 USER.md |

## 记忆系统的协同工作

一次完整对话回合中，两层记忆的协作流程：

```
用户输入 "帮我查一下明天天气"
        │
        ▼
1. 组装系统提示词
   build_enhanced_prompt()
   → 读取 USER.md (用户自述)
   → 读取 MEMORY.md (记忆叙事)
   → 拼接 AGENTS.md + SOUL.md + USER.md + MEMORY.md
        │
        ▼
2. Agent 推理 → 执行工具 → 回答
        │
        ▼
3. 回合结束 → update_narrative()
   裸消息 + 旧 MEMORY.md → LLM → 新 MEMORY.md
```

这样，Agent 在本次对话中使用了过去的记忆（通过叙事注入），同时本次对话的关键信息又被 LLM 融合进叙事，供下次使用——形成一个自我改进的闭环。
